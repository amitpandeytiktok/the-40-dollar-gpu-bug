I Set $40 on Fire in 85 Minutes — on One GPU Bug

Twelve A100s. Eighty-five minutes. A 14-billion-parameter video model. Zero clips saved. Here's the autopsy of a spectacularly dumb render failure — and the six-line checklist that makes sure neither of us ever pays for it twice.

A real incident from an AI song-video pipeline · written up the same afternoon it happened

I make music and AI music-videos. The render in question was SWAAHA — a cinematic image-to-video pass: 12 keyframes → 12 animated clips via Wan2.2 I2V-A14B on Modal, fanned out across 12 A100-80GB GPUs so they'd all finish in roughly the wall-time of a single clip.

The math is seductive: parallel = fast. The footnote nobody reads: parallel also means you can waste money twelve ways at once.

The timeline

  • 12:00 — Kick off render.starmap(). 12 containers spin up cold, each downloads the ~60–80 GB model.
  • 12:40 — Sampling begins. tqdm bars crawl forward at ~71 s/step, 40 steps to go.
  • 13:25 — 40/40 steps done on every clip. The expensive part is finished. I'm mentally already editing the video.
  • 13:26 — CUDA out of memory — inside the VAE decoder, on the very last operation: turning the finished latents into actual pixels.
  • 13:26 — That single exception trips starmap's fail-fast. All 11 healthy containers get a cancel signal mid-flight.
  • 13:26 — 0 clips saved. Eighty-five minutes of GPU time, gone, with nothing on disk.

Bug #1 — the memory cliff is at the end, not the middle

Everyone benchmarks whether sampling fits in VRAM. It did — comfortably, for all 40 steps. But a video VAE decodes all 81 frames at once, and that decode is the single most memory-hungry moment in the whole pipeline. It only happens after you've already paid for sampling. So the model sailed through the costly 99% and faceplanted on the final 1%:

# diffusers/models/autoencoders/autoencoder_kl_wan.py
OutOfMemoryError: CUDA out of memory. Tried to allocate 1.32 GiB.
GPU 0 has a total capacity of 79.25 GiB of which 1.06 GiB is free.
Process 1 has 78.18 GiB memory in use.

1.32 GiB short. On an 80 GB card. At the finish line. The fix is one of the oldest tricks in the book — decode in tiles and frame slices so peak memory stays bounded:

  pipe.to("cuda")
+ # the VAE decode of 81 frames @720p peaks >80GB — tile + slice it
+ pipe.vae.enable_tiling()
+ pipe.vae.enable_slicing()

Plus a fragmentation guard, because "1.06 GiB free but can't allocate 1.32" screams fragmentation: PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True.

Bug #2 — one death sentenced eleven innocents

Here's the part that turned an annoying bug into an expensive one. The batch was a plain fan-out:

for out_p, data in zip(todo, render.starmap(args)):   # fail-fast
    out_p.write_bytes(data)

starmap defaults to fail-fast: the instant any one input raises, the whole map is cancelled and the exception propagates. So clip #7's OOM didn't just lose clip #7 — it sent Successfully canceled input to the eleven clips that were seconds from done. Eleven healthy renders, executed and abandoned. The fix is one keyword:

+ # one clip's failure must NEVER cancel the others
ok = 0
for out_p, data in zip(todo, render.starmap(args, return_exceptions=True)):
    if isinstance(data, Exception):
        print(f"FAILED {out_p.name}: {data}"); continue
    out_p.write_bytes(data); ok += 1                # save the survivors

With return_exceptions=True, a single OOM would have cost me 1 clip, not 12. Same bug, ~$3 of damage instead of ~$40.

Why this was genuinely foolish (not just unlucky)

Bad luck is the GPU dying. This wasn't luck — it was four avoidable decisions stacked on top of each other:

I scaled before I validated. I sent twelve $3 jobs to find a bug that a single $3 job would have surfaced. The failure lived in the cheapest-to-test, last-to-run part of the pipeline — the perfect candidate for a smoke test, and I skipped it because the code "looked done."

I collected results at the end. Nothing was written until everything finished, so a late failure had maximum blast radius. Stream results to disk and there's nothing to lose.

I never put a number on the downside. 12 × $2.50 × 1.4h ≈ $42 is a ten-second calculation I did after the fact instead of before.

The 6 guardrails (steal these)

1

Canary before you fan out

Run one unit end-to-end — including the last step — before launching N. A --limit 1 flag is the highest-ROI line of code in any batch job.
2

Find the peak-memory moment, not the average

For video/image diffusion that's the VAE decode, and it's at the end. enable_tiling() + enable_slicing() by default; test at your real resolution and frame count.
3

Isolate failures in any fan-out

return_exceptions=True (Modal) / return_exceptions=True (asyncio.gather) / --keep-going. One unit's death must never cancel the swarm.
4

Save incrementally & make it resumable

Write each result the instant it lands. Re-running should skip finished work, so a crash costs minutes, not the whole batch.
5

Do the cost math out loud — first

$/GPU-hr × parallelism × wall-time. Say the number before you hit enter. Set a spend cap and an alert so the platform stops you when you won't stop yourself.
6

Pay the cold-start tax once

Twelve cold containers each re-downloaded the same 60–80 GB model — that's most of the bill. Cache weights in a shared Volume (and bake into the image) once, then scale out warm.
The cheapest place to find a bug is a $2 canary, not a $40 swarm. Test the last mile first.

The receipt, rewritten

The model is now cached in the Volume, the decode is tiled, and the batch saves survivors. The exact same render, redone correctly, costs about $25 for all twelve clips — or $2–4 to prove it on one before committing. The bug is fixed in four lines. The lesson is the asset; the $40 was just tuition.
